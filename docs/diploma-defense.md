# Защита диплома: как рассказывать про gcloude

Краткий сценарий выступления и шпаргалка по архитектуре платформы **gcloude** — PaaS для деплоя приложений из Git (Python / Node.js) с кабинетом, SSL и биллингом.

**Продакшен:** `https://app.kostricyn.ru`  
**Примеры приложений:** `examples/python-api`, `examples/node-web`

---

## 1. Введение (1–2 минуты)

**Что сказать:**

> Разработана платформа-as-a-Service для малого бизнеса и команд разработки. Пользователь регистрируется в веб-кабинете, создаёт проект, указывает Git-репозиторий — платформа сама клонирует код, собирает Docker-образ и публикует приложение на поддомене с HTTPS.

**Акцент:**

- Проблема: деплой «руками» долгий (сервер, nginx, SSL, контейнеры).
- Решение: один интерфейс + автоматизация.
- Стек: Nuxt 4 (фронт), FastAPI (бэкенд), Docker, nginx, Let's Encrypt.

---

## 2. Фронтенд — простым языком (3–5 минут)

Начните с UI: комиссии чаще всего понимают «что видит пользователь».

### 2.1. Что это такое

**Nuxt 4 + Vue 3 + TypeScript** — одностраничное приложение (SPA/SSR), которое работает как **личный кабинет оператора**.

Структура папок:

```
frontend/app/
  pages/          — маршруты (URL → экран)
  common/         — бизнес-экраны (проекты, приложения, биллинг)
  shared/         — кнопки, карточки, layout, хуки
  layouts/        — оболочка: sidebar + header
  middleware/     — проверка «залогинен ли пользователь»
```

### 2.2. Как пользователь ходит по кабинету

| Экран | Путь | Зачем |
|-------|------|--------|
| Вход / регистрация | `/login` | Email + пароль, галочки согласий |
| Дашборд | `/dashboard` | Сводка: проекты, деплои |
| Проекты | `/projects` | Группировка приложений |
| Карточка проекта | `/projects/:id` | Git URL, деплой frontend/backend |
| Приложения | `/applications` | Все задеплоенные сервисы |
| Настройки | `/settings` | Домены, профиль, юрлицо |

**Как объяснить на защите:**

> Фронтенд не хранит бизнес-логику деплоя. Он отправляет REST-запросы на `/api/v1/...`, получает JSON и рисует интерфейс. Это классическое разделение: UI отдельно, логика на сервере.

### 2.3. Авторизация и cookie

- После входа сессия сохраняется в cookie `platform_session` (7 дней, `SameSite=Lax`, на HTTPS — `Secure`).
- Middleware `auth.ts` не пускает в кабинет без сессии.
- При первом визите — **баннер cookie** (`gcloude_cookie_consent`): «только необходимые» или «принять все».
- Юридические страницы: `/legal/terms`, `/legal/privacy`, `/legal/personal-data`, `/legal/cookies`.

**Фраза для комиссии:**

> Реализовано informed consent: пользователь явно принимает соглашение и политику ПДн при регистрации, плюс отдельный баннер по cookie — это соответствует практике 152-ФЗ и GDPR-подобным требованиям к прозрачности.

### 2.4. API-клиент

Хук `usePlatformApi()` — обёртка над `fetch` к same-origin `/api/...` (nginx проксирует на gateway). Фронт не знает внутренних портов микросервисов.

## 3. Бэкенд (2–4 минуты)

### 3.1. Gateway — «мозг» платформы

**`gateway-service`** (FastAPI):

- REST API кабинета: `/api/v1/platform/*` (логин, настройки, дашборд)
- Hosting API: `/api/v1/hosting/*` (проекты, приложения, деплой)
- Прокси auth: `/api/v1/auth/*` → `auth-service`
- **Git deploy:** clone → Dockerfile → `docker build` → `docker run`
- **nginx:** пишет конфиг поддомена, **certbot** — SSL на каждое приложение

### 3.2. Остальные сервисы (MVP / задел)

| Сервис | Роль |
|--------|------|
| `auth-service` | Аутентификация пользователей приложений |
| `billing-service` | Счета, экспорт в 1С |
| `postgres` | Метаданные проектов и деплоев |
| `redis`, `kafka` | Очереди и кэш (масштабирование) |

**На защите можно сказать:**

> Архитектура микросервисная: gateway оркестрирует деплой, остальные сервисы изолированы. Для MVP критичен gateway + postgres; остальное — путь к Kubernetes.

## 4. Инфраструктура и деплой платформы (2–3 минуты)

### 4.1. Docker Compose на VPS

```
Пользователь → nginx (80/443) → frontend (кабинет)
                              → gateway-service (API)
                              → userapp-<slug> (контейнер клиента)
```

- **`setup.sh`** — установка с нуля: Docker, env, SSL, coexistence с внешним nginx.
- Домены: `app.kostricyn.ru`, `api.kostricyn.ru`, `<slug>.apps.kostricyn.ru`.
- Сертификаты Let's Encrypt: отдельно для кабинета и **для каждого приложения**.

### 4.2. Сценарий Git-деплоя (главная фича)

1. Пользователь создаёт приложение, вставляет Git URL, выбирает runtime (`node` / `python`).
2. Gateway клонирует репозиторий в `/workspace/<slug>`.
3. Если нет Dockerfile — генерируется автоматически.
4. Собирается образ `platform-userapp-<slug>`, контейнер в сети `gcloude_platform`.
5. В `platform.d/` пишется nginx-конфиг, запрос SSL, reload nginx.
6. Приложение доступно по `https://<slug>.apps.kostricyn.ru`.

**Демо на защите:**

1. Показать `examples/node-web` или `examples/python-api` в GitHub.
2. В кабинете → проект → новое приложение → деплой.
3. Открыть URL, `/api/health`, логи контейнера в карточке приложения.

## 5. Безопасность и правовые аспекты (1 минута)

- HTTPS (Let's Encrypt), прокси nginx, изоляция контейнеров в Docker-сети.
- Согласия при login/register (`accept_policy`, `accept_personal_data`).
- Cookie-баннер + политика cookie.
- Postgres/redis не проброшены наружу на VPS (только внутри Docker).

|-----|----------|-----------------|
| 1 | Открыть `app.kostricyn.ru` | «Кабинет на Vue/Nuxt» |

## 7. Возможные вопросы комиссии

**Почему Docker, а не сразу Kubernetes?**  
MVP быстрее на Compose; в ADR заложен переход на K8s + Ingress + cert-manager.

**Как масштабировать?**  
Несколько реплик gateway, вынести build в CI, registry образов, K8s Deployments.

**Где хранятся пароли?**  
Платформенный login — mock/задел под auth-service; для production — хеш в БД, JWT.

**Чем отличается от Heroku/Vercel?**  
Self-hosted PaaS под свой домен, биллинг/1С, полный контроль на VPS.

**152-ФЗ?**  
Политика ПДн, согласие при регистрации, cookie-политика, возможность обращения в поддержку.

## 9. Тезисы для заключения

- Реализован **полный цикл**: регистрация → проект → Git → контейнер → HTTPS-домен.  
- **Фронтенд** — современный стек (Nuxt 4, TS), UX кабинета, legal/cookie compliance.  
- **Бэкенд** — микросервисы, gateway оркестрирует Docker и nginx.  
- **Инфра** — one-command install (`setup.sh`), работа на реальном VPS.  
- Направления развития: K8s, очереди сборки, полноценная auth-БД, мониторинг.

```bash
./setup.sh logs            # gateway / nginx

```
